# On Termux (Android) specifically: run this FIRST, before pip install:
#   pkg install python-cryptography
#   pkg install -y clang rust make pkg-config libffi openssl
#   export ANDROID_API_LEVEL="$(getprop ro.build.version.sdk)"
# google-genai's dependency chain pulls in packages with native Rust code:
#   - cryptography (via google-auth): Termux DOES have a prebuilt package
#     for this (python-cryptography) — install it via pkg and pip will see
#     the requirement already satisfied, skipping the source build.
#   - pydantic-core (via pydantic, a REQUIRED direct dependency of
#     google-genai itself): Termux does NOT have a prebuilt package for
#     this under any name (confirmed: `pkg search pydantic` returns
#     nothing). Prebuilt wheels from PyPI are also NOT usable — they're
#     tagged for standard Linux (manylinux/glibc), while Termux's own pip
#     only accepts Android-specific tags (confirmed via `pip debug
#     --verbose`: tags like cp313-cp313-android_24_arm64_v8a, nothing
#     "linux" at all) — this is a real, structural platform-tag mismatch,
#     not a naming issue, and no clean prebuilt-wheel path was found.
#     What DOES work: building from source with Termux's own native Rust
#     toolchain (via `pkg install rust`, NOT rustup — rustup's installer
#     doesn't recognize Termux's target triple) plus the ANDROID_API_LEVEL
#     env var, which Rust/maturin-based Android builds need to know which
#     API level to target. Confirmed working via this exact combination.
#     Expect this build to take several minutes.
# This is a well-documented, common pattern for ANY Python project on
# Termux with a dependency chain reaching pydantic and/or cryptography —
# not something specific to this project.
# On a normal Linux/Mac/Windows machine, none of this is needed — pip
# finds prebuilt wheels for these there without any extra step.
